1. Foreword

This program is an editor for Commodore 64 sprites. I have written it as my small competition for free version of SpritePad and addressing a few drawbacks of already mentioned SpritePad's free version. It is not my intention to dethrone SpritePad, especially its commercial version should be better choice anyway.

Please let me introduce these drawbacks and my solutions implemented in "Master of Sprites". The main problems are: lack of batch mode support, poor support for animation with sprite overlay or lack of support for displaying connected multisprites. There are also some minor problems like small portability issues on operating systems different than Windows or relatively old-fashioned and unhandy user interface.

The problems with portability are maybe not so big and important - SpritePad runs perfectly well on Wine after all. However personally I feel it's always good and satisfying to have similar program made in Java. In other hand I didn't made anything much better in the topic of user interface. I feel fine with overall look & feel of "Master of Sprites" but everything is pretty standard as for Java Swing application. There are many drawbacks too.

The already mentioned main problems was my biggest motivation to make this program. Master of Sprites is capable to export its project from command line interface. The preview tab of its main window contains nice animation preview with support for both overlay and multisprites. The "Overlay Distance" setting makes easy to "arrange" memory bank in the way which takes into account both animation and overlay. If you check "Overlay by Next Sprite" option then current sprite will be automatically overlayed by the sprite, further by "Overlay Distance", in the memory bank. It is just that simple.

If some buttons or options of this program are mysterious thing for you, please take a look at "Quick Guide to the GUI" chapter. If you want to know how to export files from command line then go directly to "Command Line Parameters". I think this is enough for an introduction.

2. New features in the version 1.0

The version 1.0 introduce a new feature - ghost skinning. There are 4 different ghost skinning modes designed to work with animations or overlayed sprites or both. To switch between these modes go to the View->Set Ghost Skinning dialog window and choose which one to use. The dialog is hidden also under convienient keyboard shortcut CTRL + SPACEBAR.

This application by default use "Overlay stack skinning" which is helpful to show overlayed sprites in the editor component instead of only in the "Preview" tab. "Previous in memory skinning" and "Previous in selection skinning" are probably self-explaining. They are useful in creation of animations. "Intelligent animation & overlay mode" combines "Overlay stack skinning" and "Previous in memory skinning". Additionaly it wraps memory in the portions defined by "Overlay Distance".

The "Selection" menu was somewhat rearranged - most commands you can find now in the "Commands" submenu. The "View" menu has now two new options - "Editor Grid" and "Set Ghost Skinning". In the "File" menu, you can find now a few example projects.

The "Memory" tab has now new block selection mode - try to hold ALT to select any rectangular area of memory.

In previous version 0.12 sprites was overlayed in opposive direction. Now it is changed to order more natural for machine and identical as in SpritePad, so you can easily import files exported from SpritePad. I have also modified format of attributes byte to match with SpritePad.

3. Quick Guide to the GUI

Editor Panel on the Left Side

Editor Panel is used for drawing and editing one particular sprite from current selection. At the same top of the panel you can see two opposite buttons which are used for choosing current sprite from all selected sprites. Although some options here, like background color or overlay distance, are project wide and affects all sprites in memory bank.

Buttons below of the editor view itself are the same as corresponding commands from the the sprite menu. Current color options and color picker should be obvious thing for you if you are making sprites for Commodore 64. I will maybe just highlight few things about editor view:

  • Right mouse button is used for erasing - it always puts the background color.
  • CTRL + left or right click is used for flood fill whole area of the same color.
  • Mouse wheel, if you hover cursor at editor view, will quickly change the current color option.

Multicolor option is probably obvious too, but lets talk about "Overlay by Next Sprite" and "Overlay Distance". "Overlay Distance" is a project wide option which decide how far in memory bank is the next overlayed sprite. By default it is 8 sprites further, so next layer of overlay is one row below in the "Memory" tab. "Overlay by Next Sprite" is unchecked by default and you must check it to overlay starts work.


Memory Tab on the Right Side

Here you can change the selection and view all sprites in the same order as in the C64 memory. You can also change the size of memory bank by using "Quantity" spinner below the view ("Delete" command from menu just clears currently selected sprites).

If you want to make multiple sprites selection you should hold CTRL or SHIFT key while clicking by mouse. Curious thing is how copy and paste works here - pasting fills whole selection by clipboard content, so you can easily copy one sprite and paste it over whole memory bank.


Preview Tab on the Right Side

Here you can preview your animation or overlayed sprites. There is also the "Arrangement" option useful when you want to preview a multisprite made from a few connected single sprites. Lets shortly describe also other options.

The "Count" and "Delay" options next to the player buttons. "Count" is the number of animation frames - you must change it to something above one to see your animation. "Delay" is the number of miliseconds between animation frames. It decides how fast or slow animation runs. The Default is 100ms which corresponds 10Hz so you will probably want to change it to smaller value for faster animation. Minimal value of 20ms is 50Hz.

The two buttons at the top right: one with ping-pong palette and second with lock. The ping-pong button cause animation flows in different way - it doesn't loop from beginning but instead it bounces from first frame to the last and vice versa. The lock button cause preview freeze on the current selection - if pressed you can change selection in "Memory" tab and it will not change your preview.

4. Command Line Parameters

Remember that on Microsoft Windows you must first add your Master of Sprites installation folder to the PATH enviroment variable to can easily use it from command line.


Opening Project From Command Line

MasterofSprites project_file
java -jar MasterofSprites.jar project_file

With the command above you can open some project file from the command line interface. Unfortunally it doesn't support opening multiple windows with multiple projects.


Export From Command Line

Master of Sprites offers ability to export its projects from command line as well from shell scripts or batch files. Syntax of appropriate command is showed below:

MasterofSprites -e [format] project_file [-o output_file]
java -jar MasterofSprites.jar -e [format] project_file [-o output_file]

Avaible export formats are:

  • Raw binary - default export format. If you leave the format argument empty you will export your project as the raw binary.
  • PRG file - if you want to export to a PRG file then specify load address of your output PRG file as the hexadecimal number prefixed with "0x".
  • Assembly source code - there is possible to export projects to source code of these three popular assemblers: ACME, KickAss and TMPx. Just replace the format argument with the name of one of these three assemblers. Names are case insensitive here, so both ACME or acme or even aCMe are valid names.

If you do not specify output file then Master of Sprites will just change extension of your project file to raw, prg or asm.

Unfortunally currenty is not possible to make multiple exports in one command by using wildcards like '*' or '?'. On UNIX-like operating systems you can do the same job by looping in the shell script or by using capabilities of makefiles. Worse if you are using Windows batch files - as far as I know there is no looping there. :(
